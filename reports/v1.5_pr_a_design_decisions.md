# SkinMiner v1.5 PR-A Design Decisions

Date: 2026-05-01

## Fix #2: `Record.occlusion`

### Audit evidence

Watkinson dry-run evidence:

- `reports/watkinson_dryrun.md` §3, §4, and §5: each Watkinson diagnostic reports no structured occlusion, pretreatment, or replicate-count field.
- `reports/watkinson_dryrun.md` §7: `n_silicone_occluded_main_task` is `0`, and all rows fail main-task inclusion because no structured occlusion field was captured.
- `reports/watkinson_dryrun.md` §8 and §9: the failure inventory and recommendations call out occlusion/non-occlusion/pretreatment as missing structured fields.

### Information loss being addressed

The extraction products had no typed place to preserve a donor-compartment protocol modifier. Even when source text distinguishes occluded, non-occluded, or pretreated protocols, that information had no stable schema slot and could only be buried in free-text evidence or notes. Adding `occlusion` prevents this protocol state from being dropped between extraction, assembly, verification, and downstream audit.

### Why this is not a hack

The field is a closed `Literal` with an explicit `"unknown"` default. It does not infer values, rewrite existing records, or special-case Watkinson. It only extends the canonical record contract so future extractor work can populate a protocol modifier consistently and old JSONL remains valid.

### What's deferred

PR-A does not make any extractor, assembler, or patcher populate `occlusion`. It also does not add a richer protocol object, replicate parsing, or temperature/receptor completeness checks. Those are PR-B/PR-C or later scope.

## Fix #3: v4 Unit-Driven Effective Kind

### Audit evidence

Watkinson dry-run evidence:

- `reports/watkinson_dryrun.md` §4 and §5: records report `endpoint.kind` as `amount_per_area` while J units are `ug/cm2/h`.
- `reports/watkinson_dryrun.md` §8: `endpoint.kind` was usually normalized to `amount_per_area` despite flux units.
- `reports/watkinson_dryrun.md` §9: the recommendation explicitly says verification should treat flux units as flux endpoint semantics.

### Information loss being addressed

The unit string carried endpoint semantics that could be lost after extraction if `kind` was misclassified as `amount_per_area`. Under v4, flux/Jss endpoints are intentionally in scope, so verifier decisions should use the unit signal as a local effective kind while preserving the raw extracted kind for traceability.

### Why this is not a hack

The change is systematic and policy-scoped: v4 defines a closed flux-unit family, normalizes units case-insensitively, and exposes `effective_endpoint_kind()` only for records where `kind == "amount_per_area"` and the unit belongs to that family. The original `record.endpoint.kind` is not mutated. The verifier records the decision through `provenance.metadata["verification_effective_kind"]` and `provenance.metadata["verification_reclassification_reason"]`, so audit can see when the unit corrected the effective policy interpretation.

### What's deferred

PR-A does not broaden v1/v2/v3 behavior, does not add regex expansion for arbitrary unit spellings, and does not repair numeric table extraction, ternary composition parsing, or row/column alignment. It also does not change extractor prompts to emit `kind="flux"` directly; that remains future extraction work.
