# SkinMiner v1.5 PR-C pre-change observations

Date: 2026-05-10

This report was created before any PR-C code edit or table prompt rewrite.

## Current schema surface

- Canonical records still use a single endpoint field:
  - `schemas/models.py:39` defines `EndpointSpec`.
  - `schemas/models.py:135` defines `Record.endpoint: EndpointSpec`.
- `EndpointSpec` fields are:
  - `field_name`
  - `kind`
  - `value`
  - `unit`
  - `time_value`
  - `time_unit`
  - `normalized_value`
  - `normalized_unit`
- Current endpoint kind enum is:
  - `amount_per_area`
  - `amount_total`
  - `percent`
  - `flux`
  - `jss`
  - `unknown`
- `schemas/__init__.py` exports `EndpointSpec`.

## Current table extraction schema and LLM boundary

- `extractors/table/extractor.py:83` defines the table LLM row model as `FormulationRow`.
- `FormulationRow` currently has one endpoint slot:
  - `endpoint_value`
  - `endpoint_unit`
  - `endpoint_time_value`
  - `endpoint_time_unit`
  - `endpoint_kind`
- `extractors/table/extractor.py:114` defines `TableExtractionResult` with `formulations: list[FormulationRow]`.
- `extractors/table/extractor.py:600` builds the table prompt.
- `extractors/table/extractor.py:641` calls `parse_structured(..., text_format=TableExtractionResult, timeout=90)`.
- The current prompt explicitly says to prefer cumulative amount when amount and flux/Jss columns coexist. PR-C must replace this with pair-aware multi-endpoint instructions.

## Current table extractor flow

- PDF table extraction uses PyMuPDF/XML/HTML based evidence windows in `extractors/table/extractor.py`; no pdfplumber/camelot dependency is present.
- `_wide_timepoint_rows_from_window` at `extractors/table/extractor.py:771` creates deterministic Kallakunta-style rows, but each row still has exactly one endpoint value/kind/unit/time.
- `_map_formulation_to_record` at `extractors/table/extractor.py:1009` converts a `FormulationRow` to canonical `Record`.
- The mapper currently constructs `EndpointSpec(...)` at `extractors/table/extractor.py:1144`.

## Record identity and reverse-regression risk

- `utils/io.py:76` defines `make_record_id(paper_id, route, formulation_label, suffix)`.
- Table records currently add endpoint time and endpoint value into the suffix before hashing (`extractors/table/extractor.py:1096-1107`).
- Therefore `record_id` is not stable across the endpoint-list migration and cannot be used for reverse-regression equality.
- PR-C reverse validation should use semantic identity instead of `record_id`:
  - `paper_id`
  - normalized `formulation.label`
  - primary endpoint kind after old-to-new mapping
  - primary endpoint mean rounded to a stable decimal precision
  - record duration/end-time in hours rounded to a stable decimal precision
- The prompt's suggested four-tuple needs the duration/time component for Kallakunta because each formulation has multiple cumulative amount time points.

## Downstream single-endpoint dependencies

- `utils/io.py:109-116` flattens `record.endpoint` into CSV endpoint columns.
- `assembly/assemble_records.py` reads and mutates `record.endpoint` for support scoring, duration normalization, amount normalization, merge logic, record keys, and table partial filtering.
- `patchers/patch_endpoint.py` patches `endpoint.value`, `endpoint.unit`, and `endpoint.kind`.
- `patchers/patch_endpoint_time.py` patches `endpoint.time_value` and `endpoint.time_unit`.
- `policies/v1_strict_ibuprofen_5pct.py` uses `record.endpoint.value/kind/time_value`.
- `policies/v4_accept_flux.py` uses endpoint kind/unit/field name and reclassifies unit-implied flux.
- `verification/verify_records.py` reads and mutates endpoint kind/value/unit/time/normalization and stores PR-A effective-kind metadata.
- `verification/source_binding_guard.py` and `verification/llm_adjudicate.py` also read `record.endpoint`.

## Baseline JSONL schema samples

- `outputs/full_run_16_post_all_fixes/*.jsonl` uses old schema records with a nested `endpoint` object and no `endpoints` list.
- Example old endpoint shape from `verified_records.jsonl`:
  - `record_id=record_504c0645a368`
  - `route=table`
  - `endpoint.kind=amount_per_area`
  - `endpoint.value=0.5`
  - `endpoint.unit=mg/cm2`
  - `endpoint.time_value=0.0`
  - `endpoint.time_unit=days`
  - `endpoint.normalized_value=500.0`
  - `endpoint.normalized_unit=ug/cm^2`
- The old schema must fail after PR-C because `Record.endpoint` is removed and `Record.endpoints` becomes the canonical field.

## PR-A/PR-B compatibility expectations

- PR-A added occlusion tags and V4 effective endpoint kind metadata. PR-C must preserve those metadata keys and must not drop occlusion fields during migration or verification.
- The old PR-A compat test that expects `Record(endpoint=...)` should fail after PR-C. The required compatibility gate becomes deterministic migration plus v2/v4 verification on migrated JSONL.
- PR-B introduced `run_pipeline.py --input-pdf-csv`, PDF metadata cache handling, and `ContentAccess.status='success'`/`backend`. PR-C changes must not alter these surfaces.
