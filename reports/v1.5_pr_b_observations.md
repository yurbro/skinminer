# SkinMiner v1.5 PR-B Observations

Date: 2026-05-10

## 1. `utils/llm_client.py` Interface

The main provider-neutral entry point is:

```python
parse_structured(
    *,
    provider: str | LLMProvider | None = None,
    model: str,
    input: list[dict[str, Any]],
    text_format: type[SchemaT],
    timeout: float | int | None = None,
    **kwargs: Any,
) -> ParsedLLMResponse[SchemaT]
```

`LLMProvider.parse()` dispatches `"openai"` and `"anthropic"`. OpenAI calls `OpenAI().responses.parse(...)` with `text_format`; Anthropic converts OpenAI-style input messages into `system` plus `messages`, then uses `client.messages.parse(..., output_format=text_format)` or a tool fallback for nested schemas.

The interface supports separate system and user prompts because callers pass OpenAI-style messages. It supports structured output through Pydantic models, not a raw JSON-mode flag. The returned wrapper exposes `output_parsed`, `raw_response`, `provider`, `model`, and normalized token usage.

Typical existing call examples:

```python
response = parse_structured(
    provider=provider,
    model=model,
    input=[
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_prompt},
    ],
    text_format=LLMTriageResult,
)
```

```python
response = parse_structured(
    provider=provider,
    model=model,
    input=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": user_prompt},
    ],
    text_format=RouterLLMResponse,
    timeout=60,
)
```

## 2. Existing `corpus.csv` Schema

`outputs/full_run_12_full/corpus.csv` has these columns:

```text
source,id,pmid,pmcid,doi,title,abstract,year,journal,authors,url,query_used
```

`corpus/build_epmc.py` builds this by querying Europe PMC, reading core result fields, normalizing title/abstract whitespace, lowercasing DOI, coercing year to nullable integer, sorting by DOI/year, and dropping duplicates by DOI and title/year.

Key field mapping:

- `source`, `id`, `pmid`, `pmcid`, `doi`, `title`, `year`, `journal`, `authors` come directly from Europe PMC result fields.
- `abstract` comes from `abstractText`.
- `url` prefers DOI URL, then PMID URL, then the first full-text URL.
- `query_used` stores the effective Europe PMC query string.

PR-B should construct the same column set for user PDFs so rule filter, optional LLM triage, router, and later stages do not depend on whether the paper came from Europe PMC or a user upload.

## 3. `run_pipeline.py` Entry Point

`--input-csv` is currently defined with default `data/corpus_ibuprofen.csv`. `_resolve_corpus_mode(args.input_csv, args.build_corpus)` decides whether to load this CSV/JSONL or build a Europe PMC corpus if the input is missing or `--build-corpus` is set.

The corpus stage writes `outputs/<run>/corpus.jsonl` and optionally `corpus.csv`. The rule filter consumes `corpus_rows`; optional LLM triage consumes rule-pass rows; access consumes `triaged_rows`; `_build_content_handles()` and `_build_route_inputs()` prepare router inputs from `ContentAccess`.

The minimal PR-B insertion point is:

- argparse: add mutually exclusive `--input-csv` / `--input-pdf-csv`, plus `--refresh-metadata`.
- corpus stage: if `--input-pdf-csv` is present, read the PDF CSV, extract/cache metadata, and build standard corpus rows.
- access stage: if `--input-pdf-csv` is present, write final local-PDF `ContentAccess` records instead of calling OA download/lookup logic.

This avoids changing router, extractors, patchers, verifier, or policy behavior.

## 4. Access Stage

`access/resolve_content.py` resolves DOI/PMID through Europe PMC and Unpaywall, optionally materializes local artifacts, and returns `ContentAccess` with:

```python
paper_id, doi, pmid, pmcid, title,
preferred_format, available_formats,
access_urls, local_paths, status, notes
```

Input rows are corpus-like mappings. Output is JSONL/CSV of `ContentAccess`. Current status values are `resolved`, `downloaded`, `unresolved`, and `error`; there is no explicit backend field yet.

For user-uploaded PDFs, the cleaner approach is to short-circuit at the pipeline access stage: construct `ContentAccess(local_paths={"pdf": <user_pdf_path>}, preferred_format="pdf", available_formats=["pdf"], status="success", backend="user_upload")` and write it to the normal `content_access.jsonl`. The OA resolver then remains unchanged for existing `--input-csv` runs.

## 5. PDF Text Extraction Dependency

The repository already uses PyMuPDF (`fitz`) in router, table extraction, text page selection, figure extraction, patchers, and single-pass experiments. `requirements.txt` only lists `openai` and `anthropic`, so PyMuPDF is an environment dependency already assumed by the current codebase.

PyMuPDF is suitable for reading the first one or two text pages of normal scientific PDFs. It will not OCR scanned-image PDFs; PR-B should treat empty/unparseable first-page text as a graceful metadata extraction failure/fallback, not as a reason to add OCR.

## 6. Out-of-Scope Observations

- `ContentAccess` lacks a `backend` field and `success` status. Adding these to the access handle schema is useful for auditability, but no router/extractor/verifier policy logic should depend on them.
- The validation command in the task references `--stop-after-stage router`; the current CLI does not expose such a flag. A small run-pipeline stop point would make validation deterministic and avoid running extraction during PR-B validation.
