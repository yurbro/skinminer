# SkinMiner v1.5 PR-B Design Decisions

Date: 2026-05-10

## Fix #7a: `run_pipeline.py --input-pdf-csv`

### Audit evidence

`reports/watkinson_dryrun.md` documented that SkinMiner had no formal local PDF entry point. The dry-run had to manually construct a minimal corpus and `ContentAccess(local_paths={"pdf": ...})` before invoking later stages.

### Information loss being addressed

Before PR-B, a user-owned PDF could only enter SkinMiner through an ad hoc workaround. That meant bibliographic metadata, local PDF provenance, and cache state were not preserved by the same run manifest and stage outputs as the Europe PMC path. The first information loss happened before scientific extraction: the framework could not represent that a paper was already available locally.

### Why this is not a hack

The new CLI path keeps the same downstream contract. User PDFs are converted into standard corpus rows, then into standard `ContentAccess` records. Rule filtering, optional LLM triage, and routing do not receive a special Watkinson-only branch.

### Deferred items

PR-B does not add single-PDF CLI syntax, drag-and-drop UX, or online metadata fallback.

## Fix #7b: LLM Metadata Extractor

### Audit evidence

User-uploaded PDFs need title, DOI, year, journal, authors, and abstract before they can pass through the existing rule filter and router. The repository already had a provider-neutral structured LLM interface in `utils/llm_client.py`, but no PDF metadata extractor.

### Information loss being addressed

Without a metadata extractor, the framework would either lose the abstract entirely or require users to manually retype bibliographic fields into a corpus CSV. That loses the connection between the local PDF artifact and the metadata used to screen and route it. The new extractor makes the metadata capture explicit, cacheable, and auditable.

### Why this is not a hack

The extractor reads only the first two PDF pages with the existing PyMuPDF dependency, asks only for bibliographic metadata, and uses `parse_structured(...)` rather than hard-coded OpenAI or Anthropic calls. It does not extract methods, tables, endpoints, or scientific results. Those remain owned by the existing downstream stages.

### Deferred items

PR-B does not add OCR for scanned PDFs, CrossRef fallback, metadata reconciliation, or confidence scoring beyond warnings and cache inspection.

## Fix #7c: Metadata Caching

### Audit evidence

The task required reproducible repeated runs for the same PDF. The implemented cache path is the sibling JSON `<pdf_path>.metadata.json`.

### Information loss being addressed

LLM-derived metadata can vary across calls. Without a cache, the same user PDF could produce a different abstract string or title normalization on a later run, changing rule-filter and router inputs. The cache freezes the paper-level metadata at the first successful extraction, while `--refresh-metadata` gives an explicit opt-in path to regenerate it.

### Why this is not a hack

The cache is local to the source artifact and contains the extraction method, model, source PDF hash, warnings, and fields used to build the corpus row. It is not hidden global state and does not mutate the frozen v1 corpus.

### Deferred items

PR-B does not implement cache version migration or merge conflict tooling for hand-edited metadata caches.

## Fix #7d: Internal Minimal Corpus Construction

### Audit evidence

`outputs/full_run_12_full/corpus.csv` uses the columns `source,id,pmid,pmcid,doi,title,abstract,year,journal,authors,url,query_used`. `corpus/build_epmc.py` constructs those fields from Europe PMC core records.

### Information loss being addressed

If PDF metadata were passed through a separate object shape, each downstream stage would need to know about user uploads and could accidentally drop fields that the Europe PMC path always has. PR-B converts PDF metadata into the existing corpus shape so the screening and routing stages consume a stable paper-level contract.

### Why this is not a hack

The user PDF corpus rows are generated in memory and written only to the run output directory. They do not alter `data/corpus_ibuprofen.csv` or any frozen baseline output.

### Deferred items

PR-B does not add a permanent user corpus registry. Users can keep the generated run-level `corpus.csv` or the metadata caches as the reproducible source of record.

## Fix #7e: Access Stage Local PDF Short-Circuit

### Audit evidence

`access/resolve_content.py` is built around DOI/PMID lookup and optional OA download. For user PDFs, that work is unnecessary and can introduce failure modes unrelated to the supplied artifact.

### Information loss being addressed

The important access fact for this path is that the PDF is already local. PR-B preserves that fact in `ContentAccess.local_paths["pdf"]`, with `status="success"` and `backend="user_upload"`, so downstream stages can use the same access handle shape while audit can distinguish user-uploaded artifacts from OA-resolved artifacts.

### Why this is not a hack

The short-circuit lives in the pipeline access branch for PDF input mode. Existing `--input-csv` runs still call `resolve_content_batch(...)` and use the original OA resolver behavior.

### Deferred items

PR-B does not add mixed input mode combining Europe PMC corpus rows and user PDFs in one run.

## Workaround Cleanup

The dry-run script `outputs/watkinson_dryrun/_run_pipeline_stages.py` was moved into `_v1_backup_pr_b_2026-05-10-0058/legacy_workaround/_run_pipeline_stages.py`. The formal entry point is now `run_pipeline.py --input-pdf-csv ...`.

## Out-of-Scope Observations

1. The router still decides route quality; PR-B only ensures user PDFs reach it.
2. Scanned PDFs without extractable text will need manual metadata hints or future OCR support.
3. A richer `ContentAccess.backend` enum may be useful later, but PR-B keeps it as a string for backward compatibility.
