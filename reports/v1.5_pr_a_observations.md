Because `Record` uses `extra="forbid"` and the new field will have `default="unknown"`, old JSONL rows that omit `occlusion` should remain loadable. The required post-change compatibility script will confirm every row loads and receives `occlusion == "unknown"`.

Post-change confirmation: the temporary script `scripts/verify_pr_a_schema_compat.py` loaded `239/239` rows from `outputs/full_run_16_post_all_fixes/verified_records.jsonl`, with `failed=0`; every loaded row had `occlusion == "unknown"`. The temporary script was removed after the check.

### 6. Verifier entry point and current decisions

The main verifier entry point is `verify_records()` in `verification/verify_records.py:905`.

For each input record it:

- creates a deep copy named `candidate` (`verification/verify_records.py:924`)
- normalizes fields via `_normalize_record_fields()` (`verification/verify_records.py:925-932`)
- computes evidence support (`verification/verify_records.py:933`)
- collects policy and evidence failure codes via `_collect_failure_codes()` (`verification/verify_records.py:934`)
- sets `verification_status` with `classify_outcome()` (`verification/verify_records.py:936`)
- derives scope bucket/tags (`verification/verify_records.py:937`)
- writes JSONL/CSV outputs via `write_records_jsonl()` and `write_records_csv()` (`verification/verify_records.py:946-949`)

The endpoint unit is read from `record.endpoint.unit`; the endpoint kind is read from `record.endpoint.kind`. v4 currently accepts flux-like units in `V4AcceptFluxPolicy.endpoint_scope_status()` (`policies/v4_accept_flux.py:74-83`), but later generic failure-code logic still checks the original kind at `verification/verify_records.py:859`, `verification/verify_records.py:870`, and `verification/verify_records.py:873`. This is the narrow location where v4-only `effective_kind` handling should prevent flux-unit records from being treated as failed `amount_per_area` normalization.

### Per-record reclassification output choice

The verifier currently outputs `Record` objects only; there is no separate per-record verification-output object. For PR-A, the least disruptive audit channel is `candidate.provenance.metadata`, adding keys only when v4 unit-driven reclassification occurs:

- `verification_effective_kind`
- `verification_reclassification_reason`

This preserves the original `record.endpoint.kind`, avoids schema expansion beyond `occlusion`, and makes the reclassification visible in `verified_records.jsonl`.
